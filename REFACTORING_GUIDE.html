1 왜 바꿨나요? — 문제 정의

❌ 기존 방식의 문제점
  • State가 Component를 상속하여 매 프레임 Update() 호출
  • 다른 Component가 State*를 캐시해 직접 폴링 (IsDead(), IsAttacking())
  • State 변경 시점을 아무도 모름 → 반응 타이밍 불명확
  • State 로직 + 타이머 + 조건 판단이 한 파일에 혼재
  • 새 Component가 State를 참조하려면 매번 직접 연결 코드 추가
✅ 새 방식의 목표
  • State = 순수 데이터 보유자, 값만 들고 있음
  • 값이 바뀌면 자동으로 구독자에게 통보 (Observer Pattern)
  • 반응 로직은 Callbacks 폴더에 중앙 집중
  • 공격 타이머 등 로직은 AttackController로 분리
  • Component들은 State를 폴링하지 않음

4 Observable State 패턴

클래스 계층도

ObservableState<TEnum> 구현

// FrameWork/State.h
template<typename TEnum>
class ObservableState : public State {
public:
    using Callback =
      std::function<void(TEnum prev, TEnum next)>;

    TEnum Get() const { return current; }

    void Set(TEnum next) {
        if (current == next) return;  // 동일 값 무시
        const TEnum prev = current;
        current = next;
        // 변경 즉시 모든 구독자 통보
        for (auto& cb : subscribers)
            cb(prev, next);
    }

    void Subscribe(Callback cb) {
        subscribers.push_back(std::move(cb));
    }

protected:
    TEnum current{};
    std::vector<Callback> subscribers;
};
Set()이 하는 일
① 이전 값과 같으면 아무 것도 안 함
② 다르면 current 업데이트
③ 모든 구독자 함수를 순서대로 호출
Subscribe()가 하는 일
콜백 함수를 내부 배열에 추가.
이후 Set()이 호출될 때마다 자동 실행됨.
구독 해제는 지원하지 않습니다
GameObject 수명과 동일하게 State가 살아있으므로, Component 소멸 시 이미 State도 소멸됩니다.

6 각 레이어 역할 정리

폴더/파일 역할 Update() 있음? State 읽기 State 쓰기
States/ 열거형 값을 보유하고, 변경 시 구독자에게 통보 없음 자기 자신만 Set() 내부
Callbacks/StateCallbacks State 변경 시 어떤 Component 메서드를 호출할지 결정 없음 ComputeClipName에서 읽음 안 씀
AttackController 공격 타이머 관리, Set(SwordAttack) / Set(NoAttack) 호출 있음 (타이머 감소) 안 읽음 Set() 호출
PlayerControl 입력 처리 → MovementState 업데이트, 공격 트리거 있음 (입력 폴링) isMovementLocked / isAttackLocked 플래그만 MovementState.Set()
SpriteAnimator 클립 재생, 프레임 전진 있음 (프레임 타이머) 안 읽음 안 씀
Core/main.cpp 오브젝트 조립, AddState / AddComponent 순서 관리 없음 없음 없음
등록 순서가 중요합니다 (main.cpp)
States → AttackController → PlayerControl → VelocityController → SpriteAnimator → MeshRenderer
Start()에서 GetState<T>()를 사용하기 때문에 State가 먼저 등록되어 있어야 합니다.

7 코드 비교 — Before vs After

① SpriteAnimator::Start() — 클립 초기화 방식

Before — 직접 캐시 후 폴링
void SpriteAnimator::Start() {
    // State 포인터를 직접 캐시
    movState = pOwner->GetComponent<MovementState>();
    atkState = pOwner->GetComponent<AttackState>();
    lifState = pOwner->GetComponent<LifeState>();
    // Update()에서 매 프레임 IsDead() 등을 호출
}
After — Subscribe로 등록
void SpriteAnimator::Start() {
    auto mov = pOwner->GetState<MovementState>();
    mov->Subscribe([this](MovementStateType p, MovementStateType n){
        StateCallbacks::OnAnimMovement(this, p, n);
    });
    // Attack, Life도 동일하게 구독
    // 초기 클립 동기화
    StateCallbacks::ReevaluateAnimClip(this);
}

② PlayerControl::Update() — 죽음 판정

Before — 매 프레임 State 폴링
void PlayerControl::Update(float dt) {
    // LifeState 직접 호출
    if (lifeState->IsDead()) {
        pOwner->velocity = { 0, 0, 0 };
        return;
    }
    // AttackState 직접 호출
    if (attackState->IsAttacking()) return;
    // ... 입력 처리 ...
}
After — 플래그만 확인
void PlayerControl::Update(float dt) {
    // 콜백이 미리 세팅한 플래그만 읽음
    if (isMovementLocked) {
        pOwner->velocity = { 0, 0, 0 };
        return;
    }
    if (isAttackLocked) return;
    // ... 입력 처리 ...
}

③ main.cpp — State 등록 방식

Before
// State를 Component처럼 등록
player->AddComponent(new AttackState());
player->AddComponent(new LifeState());
player->AddComponent(new MovementState());
player->AddComponent(new PlayerControl(0));
player->AddComponent(new SpriteAnimator(&mesh));
After
// State는 별도 컬렉션에 등록
player->AddState(new AttackState());
player->AddState(new LifeState());
player->AddState(new MovementState());
// AttackController가 타이머 로직 담당
player->AddComponent(new AttackController());
player->AddComponent(new PlayerControl(0));
player->AddComponent(new SpriteAnimator(&mesh));

8 새 파일 한눈에 보기

NEW
FrameWork/State.h

모든 State의 공통 기반. ObservableState<TEnum> 템플릿 정의.

  • State (base): pOwner만 들고 있음
  • ObservableState: Get/Set/Subscribe
NEW
Components/AttackController.h/.cpp

기존 AttackState에 있던 타이머 로직을 분리. Component로서 Update()를 가짐.

  • TriggerSword(dur) / TriggerMagic(dur)
  • Update(): remainingTime 감소 → NoAttack
NEW
Callbacks/StateCallbacks.h/.cpp

State 변경 시 어떤 Component 메서드를 호출할지 결정하는 반응 함수 모음.

  • OnAnimMovement / OnAnimAttack / OnAnimLife
  • OnControlLife / OnControlAttack
  • ReevaluateAnimClip (초기 동기화)
NEW (이동)
States/MovementState.h/.cpp

Components/에서 States/로 이동. Component 상속 제거.

  • SetFromDirectionInput() 유지
  • GetStateName() / GetDirectionName()
  • Update() 없음
NEW (이동)
States/AttackState.h/.cpp

타이머 로직 제거됨 (AttackController로 이전). 데이터만 남음.

  • IsAttacking() / GetStateName()
  • remainingTime, TriggerSwordAttack 삭제
NEW (이동)
States/LifeState.h/.cpp

SetAlive() / SetDead() 가 내부적으로 base Set()을 호출.

  • IsAlive() / IsDead() 유지
  • GetStateName() / ToString() (public)
변경 규칙 요약: 새로운 State 타입을 추가하려면 ObservableState<MyEnum>를 상속하는 클래스를 States/에 추가하고, 반응 함수를 Callbacks/StateCallbacks.cpp에 추가한 뒤, Start()에서 Subscribe하면 됩니다. Component의 Update() 안에 조건문을 추가할 필요 없습니다.
